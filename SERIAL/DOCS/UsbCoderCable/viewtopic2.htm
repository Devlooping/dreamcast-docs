Just a quick update. Replacing the 6.000Mhz crystal on the USB-Serial module
with a 6.144Mhz allows transfers at 1.56K baud now. With the 6Mhz crystal
the rate error between the DC and the module was around 5% which is too
large. Using the 6.144Mhz crystal that error is now 1%. The change is large
enough to allow us to use it for the DC, yet small enough that it does not
cause problems with USB communications side. The cost of the module is $25
and is available from Mouser electronics. I still have some issues to work
out regarding power and reset on the hardware side. Also since upping the
baud to 1.56M I"m seeing some timeout issues that may be related to the
internal fifo buffers and tx timeout settings in the module. I think those
can be adjusted by re-programming some values in the on-board serial eeprom.

Oh and the statement about reaching the limits of the SCIF. Ha! such a fool
was I. I"ve re-read the Hitachi SH4 hardware manual and now see the error of
that statement

Posted: Thu Sep 16, 2004 1:37 am    Post subject: Cygwin... USB-Serial Module power issues...
Originally Posted: Mon Sep 06, 2004 2:16 pm - Cryptic Allusion Games Forum

This post is the result of some dialog between Andrew Kieschnick, Dan Potter and myself about dcload-serial maintenence. While getting ready to make a minor update to the tool I realized the need to test it under Cygwin too so...

I've never tried to build a sh4/arm toolchain under Cygwin before. After bombing a couple of times and the hour growing late I wimped out and downloaded a pre-compiled set. Maybe I'll try and figure out what I did wrong... or not.

Anyway the mods I made compile just fine since most of them are in the termios sections. There are really no mods needed for windows/cygwin as it turns out. The current version of dcload-serial- 1.0.3 along with the FTDI VCP (Virtual ComPort) driver let you specify the 500k and 1.5M baud settings just fine.

There are a few things I don't like about the way the prototype USB-Serial codes cable works:
1) The device gets it's power from the USB-bus, but to prevent damage to the DC when it's turned off I pull the device reset low which places the interface lines in tri-state. This should prevent current from flowing into the DC's serial port when the PC is on and the DC is off. The downside of this is that when the DC is off, the USB module is in reset which causes the host OS to unload the driver. Under Linux this is does not seem to be a problem. When powering up the DC under Windows ME however the system freezes for about 20-30 seconds while the USB driver gets loaded. Under Linux it's instantaneous. Under MAC???

2) One solution is to add a non-inverting tri-state buffer between the Module and the DC, then let the DC control the tri-state line of the buffer while leaving the USB module powered up and running.

3) Operating the DC with no host/or the host powered down, attached to the USB-Serial device put's the device at risk of damage from current flowing into it from the DC's 3.3v line and IO pins.

4)Ideally the USB-serial device would be powered from the DC's +5v line. With the current DC serial connector that line is not available. I have ordered a VS link cable from Lik-Sang and will ohm it out to see what lines are available. I hate the idea of hacking up a over-moulded connector to rewire it. I don't know if that connector style/model is available on the electronic connector market. This solution still has the problems stated in item 1, but solves 2 and 3.

There are some other developers working faster RS232 based cables using faster MAX converter chips and modified PC serial ports, but I think the USB-serial prototype is a bit easier to construct and get parts for. Once all the bugs a worked out that is.

Of course I could just forget about it and find a Lan Adapter Razz Probaby be cheaper in the long run.

This is the last post I made since the message loss during the YaBB upgrade. If anyone posted a reply to this I'd appreciate it if you would repost your comments.

From you statement I gather your modified serial card talks at 1.56M baud when set to 38400 baud. Since version 1.0.3 the serial versions of dc-tool and dcload by default talk at 57600 baud and then switch based on the command line args.

In your case you would probably want to compile dcload to always talk at 1.56M baud, and dc-tool at 38400 baud. The modifications are not difficult. This is just a quick ugly hack:

In dc-tool you would have to defeat the baud changing code in change_speed(...) and then undef and then define BAUD_RATE to 38400.

dc-tool.c: (1.0.3)
Code:

/* somewhere near the top of dc-tool.c add these two lines */
#undef BAUD_RATE
#define BAUD_RATE 38400
...
int change_speed(char *device_name, unsigned int speed)
{
    unsigned char c;
    unsigned int rv;

    c = 'S';
// Comment this out
//    serial_write(&c, 1);
//    blread(&c, 1);
//    if (speedhack && (speed == 115200))
//        send_uint(111600);  /* get dcload to pick N=13 rather than N=12 */
//    else
//        send_uint(speed);

    printf("Changing speed to %d bps... ", speed);
#ifdef _WIN32
    FlushFileBuffers(hCommPort);
    CloseHandle(hCommPort);
#else
    tcflush(dcfd, TCIOFLUSH);
    close(dcfd);
#endif
    if (open_serial(device_name, speed)<0)
        return 1;
//  Comment this out
//    send_uint(rv);
//    rv = recv_uint();
    printf("done\n");
}


In dcload you would undef SERIALSPEED and then define SERIALSPEED to 1500000

dcload.c (1.0.3)
Code:

/*near the top of dcload.c */
#undef SERIALSPEED
#define SERIALSPEED 1500000


Now dc-tool will operate at 38400 (1.56M on your card) and dcload will work at 1.56M baud.
